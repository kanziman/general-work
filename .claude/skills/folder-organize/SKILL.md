# Folder Organize Skill

이 스킬은 사용자가 "폴더 정리해줘", "하위 폴더 만들어줘", "지식창고 정리해줘" 등의 명령을 통해 지식창고(Vault) 내의 파일 분류 체계를 정비하고자 할 때 동작하는 스킬입니다.

## 1. 기본 정보 및 트리거 규칙
- **트리거**: "폴더 정리해줘", "하위 폴더 만들어줘", "지식창고 정리해줘" 등 폴더 및 파일 구조 정리를 요청하는 수동 입력 시 실행됩니다.
- **실행 제한**: 이 스킬은 사용자가 명시적으로 요구할 때만 작동해야 하므로 `disable-model-invocation: true` 설정이 유효해야 합니다.

## 2. 안전 규칙 (Safety First)
- **[CRITICAL]** 사용자로부터 최종 미리보기 승인을 받기 전까지는 파일의 이동이나 폴더 생성을 절대 수행하지 마십시오.
- **[권장사항]** 작업을 시작하기 전에 반드시 현재 상태가 Git에 모두 커밋/백업되었는지, 혹은 Obsidian Sync 동기화가 완료되었는지 사용자에게 묻고 확인을 권장하십시오.

## 3. 동작 절차

### [1단계] 대상 폴더 분석
1. 사용자가 지정한 상위 폴더(예: `10_보고서/`, `20_회의록/` 등) 또는 전체 Vault를 대상 폴더로 설정합니다.
2. 대상 폴더에 있는 모든 파일의 파일명과 Frontmatter(특히 `type`, `tags`, `date` 필드 등)를 분석하여 메타데이터를 수집합니다.

### [2단계] 하위 폴더 구성안 제안
1. 수집한 메타데이터를 바탕으로 가장 논리적인 하위 폴더 구조 기준 후보를 탐색합니다:
   - **시간순 기준**: 연도별(예: `/2026/`), 분기별(예: `/2026/Q2/`) 분류
   - **주제·프로젝트 기준**: Frontmatter 태그 군집 기반 분류 (예: `/멤버십/`, `/이벤트/` 등)
   - **문서 유형 기준**: 문서의 성격 기반 분류 (예: `/성과보고/`, `/기획안/` 등)
2. 대상 폴더의 성격에 적합한 **1~2개의 하위 폴더 정리 기준**을 설계하여 사용자에게 제안합니다.

### [3단계] 미리보기 및 승인 대기
1. 제안하는 폴더 구조를 아래 형식의 **구조 트리**와 **파일별 이동 표**로 작성하여 미리보기로 제시합니다.

   #### 폴더 구조 미리보기 트리
   ```text
   대상폴더/
   ├── 제안_하위폴더A/
   │   ├── 파일1.md
   │   └── 파일2.md
   └── 제안_하위폴더B/
       └── 파일3.md
   ```

   #### 파일 이동 예정 표
   | 원본 파일 경로 | 제안하는 대상 폴더 경로 | 분류 기준 요약 |
   | :--- | :--- | :--- |
   | `원본폴더/파일1.md` | `원본폴더/제안_하위폴더A/파일1.md` | 태그 기반 (#멤버십) |

2. **[STOP]** 미리보기를 출력한 뒤 즉시 진행을 멈추고 사용자의 명시적인 검토 및 승인을 대기합니다.
3. 사용자가 피드백을 통해 분류안 수정을 요구하는 경우, 이를 반영하여 다시 미리보기를 제공하고 다시 대기합니다.

### [4단계] 실행 및 후처리 (승인 후)
사용자가 최종 승인하면 다음 순서로 파일 정리를 조심스럽게 진행합니다:
1. **폴더 생성 및 이동**: 승인된 하위 폴더를 생성하고 파일을 해당 경로로 이동시킵니다.
2. **링크 복구**: 파일 이동으로 인해 깨질 우려가 있는 `[[위키링크]]`와 MOC 문서에 저장된 해당 파일의 연결 링크 및 상대 경로를 보관함 전체에서 검색하여 올바른 경로로 일괄 수정합니다.
3. **MOC 및 규칙 갱신**: `CLAUDE.md`에 기술된 폴더 구조 규칙을 새로운 하위 폴더 체계에 맞게 갱신하고, 관련 MOC 파일 내의 링크 정보도 최종 최신화합니다.

### [5단계] 최종 요약 보고
- 파일 정리가 종료되면 다음 정보를 요약하여 보고합니다:
  - 실제로 이동된 파일 목록 표 (원본 경로 -> 신규 경로)
  - 이동 후 수정/보구된 전체 위키링크 및 MOC 내 참조 링크의 총 개수
